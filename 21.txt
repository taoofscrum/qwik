　最後の一個前のスライドにある、
　DoD と 受入条件( Acceptance Criteria) に関しての「定義の違い」って知ってますか～？
　皆さんの実務上の違いはまあ、その場で議論すればいいんだけど、源流知っている方がいればぜひおしえ～て。

　DoD は Scrum だよね。
　　http://en.wikipedia.org/wiki/Scrum_(development)
　　　The exit-criteria to determine whether a product backlog item is
complete. In many cases the DoD requires that all regression tests
should be successful.

　　Potentially Shippable を決める条件リストとして説明されてる
　　　http://www.scrumalliance.org/articles/105-what-is-definition-of-done-dod
　　　http://www.scrumalliance.org/articles/106-definition-of-done-a-reference

　じゃあ Acceptance Criteria という用語はどこからきたのか？
　なんとなくPM方面かなとは思うんですけど。

　ユーザーストーリーで Acceptance Test の条件を書くというのは、
　Mike Cohrn の User Stories Applied にでてきます。

　　Chapter.6 Acceptance Testing User Stories
　　　テストすべき項目の備忘録をカードの裏に書く

　　　　例: クレジットカードで支払う事ができる
　　　　　・VISA、マスター、AMEXでテストする (Pass)
　　　　　・ダイナーズでテストする (失敗)
　　　　　・正規のカード番号、エラーな番号、存在しない番号でテストする
　　　　　・期限切れカードでテストする
　　　　　・数量の違う注文(カード上限を超えるものも含む)でテストする
　　　　...このあと Fit/FitNesse でテストを書く話に進む

　まあ、Acceptance Criteria 自体は、古来からある、一般用語だとは思うのですが。
　PMBOKには、プロジェクトのスコープ宣言、というのをのっけてるみたいです。

http://en.wikibooks.org/wiki/Project_Management/PMBOK/Quality_Management

Project Scope Statement
プロジェクトのスコープ宣言

The project scope statement details the deliverables, objectives,
thresholds and acceptance criteria that the project must meet. This
makes it very important to quality planning.
Acceptance criteria describe the requirements and conditions that must
be achieved before deliverables will be accepted. If the deliverables
satisfy the acceptance criteria, then the result is the customer's
needs being met. The acceptance criteria can drastically increase or
decrease the costs of project quality. In addition, the product scope
statement may contain a scope description which contains issues that
may affect quality planning.

プロジェクトのスコープ宣言はプロジェクトが満たすべき納品物、目的、閾値を詳細に記述する。これは品質の計画にたいへん重要である。
受け入れ条件は納品物の受け入れ前に達成すべき要求・条件を説明する。納品物が受け入れ条件を満たしているならば、その結果、顧客ニーズを満たしているトイウ事になる。受け入れ条件によってはプロジェクトの品質コストが劇的にあがったりさがったりする。また、プロジェクトのスコープ宣言は品質計画に影響する諸問題の説明を含む事がある。

　元記事をちょっと訳しました。
　　http://www.scrumalliance.org/articles/105-what-is-definition-of-done-dod

　(翻訳ここから)
　"DoD is informed by reality. (DoDは現実から得られる情報だ)"
　スクラムではチームがスプリントの終わりで "potentially shippable software
(潜在的に出荷可能なソフトウェア)"
を出す事を求めている。私にとっては、それはリリース可能な機能のことだ。あまりはっきりと注意を呼びかけなくても、プロダクトオーナーの裁量でエンドユーザに提供する事ができるものだ。そのプロダクトを２日以内にエンドユーザに届ける事ができるなら、"potentially
shippable"な状態である、としている。理想的には、"potentially shippable"
が、完了の定義(DoD)と全く同じになる。

実際、多くのチームは"potentially
shippable"な状態に向かって仕事をしている。そういったチームは、完了の定義(DoD)を複数のレベルで持っているのではないだろうか。
 - 機能の完了の定義 (ストーリー、または、プロダクトバックログ項目)
 - スプリントの完了の定義 (スプリント内で開発される機能のセット)
 - リリースの完了の定義 ( "potentially shippable" な状態 )

それぞれの完了の定義(機能、スプリント、リリース)に影響する様々な要因があるのだが、究極的には、以下の質問に答えられれば完了と判断する。
 - 各機能に関して、これ(テスト等)を実行することができるか？ (できないなら次の項目)
 - 各スプリント(の成果物)に関して、これ(テスト等)を実行することができるか？  (できないなら次の項目)
 - リリースにおいては、これ(テスト等)を、やらなきゃいけないよね!
　(翻訳終わり)

　つまり、そのタイムボックス内にできるかどうか、ということで、条件をふるいにかけているようです。
　１番目はストーリーに裏書きするか、グランドルールにして、
　２番目の質問以降はチケット化するんでしょうかね。
